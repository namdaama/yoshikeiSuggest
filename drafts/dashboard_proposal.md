# 運営向けダッシュボード提案 ─ 表示項目案（モックアップ準拠）

> **取扱注意：** 2026-04-24 電話での追加スコープに対応するドラフト。モックアップ ([dashboard_mockup.html](dashboard_mockup.html)) と整合。合意後に pptx 本体へ反映する。
>
> **前提：**
> - 読者は**運営（ヨシケイ開発の運用側）**、および分析実務担当。開発系会社のため、グラフ・チャートを用いた分析ビューで問題ない。
> - 媒体は提案側の設計（先方からの指定なし）。モックアップは静的な分析用Webダッシュボード（軽量SVGチャート、外部BIツール不使用）で作成している。
> - 素材は `楽楽web_改善効果_測定構造_算出定義付き.xlsx`（先方既定義の3層6KPI）と `楽楽web_バックエンド処理ロジック一覧.pdf`（8カテゴリ）。

## 設計方針

1. **先方既定義の6KPI を "頻度適合のビュー" に配置する** ─ 週次ビュー＝行動KPI（第1層）、月次ビュー＝関係KPI（第2層）、四半期ビュー＝結果KPI（第3層）。各ビューには**そのビューの頻度に合う主指標のみ大きく表示**し、他頻度のKPIは別ビューで扱う。
2. **業界標準の補完項目は "絞って" 入れる** ─ 運営判断に毎回使うもの（S/A判定）のみ採用。ファネル・FC別・コース別・コホート等の "運営が掘れる素材"を限定的に配置。
3. **4つの時間軸で構造化** ─ 日次／週次／月次／四半期。頻度ごとにビューを切替。
4. **バックエンドPDF ⑥⑦⑧ が主データソース** ─ ⑥行動履歴の蓄積／⑦KPI算出前処理／⑧管理・運用向け参照。既存ベンダー取りやめ方針のため、この3カテゴリはリバプレ側で設計・実装する前提。
5. **グラフ主体の分析用Web（軽量実装）** ─ 大型BIツール（Tableau／Looker）ではなく、静的HTML＋軽量JS（外部ライブラリ不使用で SVG 直描画）で実装する想定。モックアップを参照。軽量化することで実装・運用コストを抑える。

## ダッシュボード構成（4ビュー）

### ビュー A：日次（毎朝の見回り用）

運営が朝一で開いて、前日〜当日の異常検知に使う。先方KPIは効果測定主体で日次には該当しないため、運営監視指標を業界標準から最小限補完。

| 項目 | データソース（PDF該当） | 表示形式 |
|---|---|---|
| 前日注文件数（14日推移） | ③注文集計ロジック | 折れ線グラフ（当週・前週同曜日・4週移動平均の3系列重ね） |
| 本日締切コース別の現時点件数と着地予測 | ②注文状態・締切管理 | コース別テーブル（前週同時点比＋最終予測） |
| FC別アラート（閾値超過時のみ） | ③注文集計ロジック | Callout（該当FC・下落幅・疑う要因） |

**出し方：** 毎朝 07:00 前後に前日分を集計。ダッシュボード更新＋異常時のみメール／Slack通知。

---

### ビュー B：週次（改善施策の効き目を見る）

**先方第1層：行動KPI（週次）を中心に、施策効果の追跡。施策リリース日をチャート上に注釈として重ね、"いつの施策がどこから効いたか" を読み解く。**

| 項目 | 出典 | 表示形式 |
|---|---|---|
| **ログイン後直帰率** | 先方KPI（行動KPI） | KPIカード（現値＋前週比＋12週スパークライン） |
| **注文未実施週の再訪率** | 先方KPI（行動KPI） | KPIカード（同上） |
| **主指標12週チャート（施策注釈付き）** | 先方KPI（行動KPI／主指標） | 折れ線＋4週移動平均＋目標ライン＋**施策リリース日の垂直注釈**（例：W8 FV改修、W13 ポップアップ追加） |
| 週次ファネル（訪問→ログイン→カート→注文完了） | 業界標準補完 | 横棒ファネル＋前週位置ゴースト線＋離脱箇所 Signal |
| FC別実績（上位5＋アラート） | 業界標準補完 | 横棒ランキング＋前週比＋12週スパークライン。閾値超過FCは色分け |
| 分析所見（1段落） | 改善推進担当 | 次週の打ち手仮説を短く |

**出し方：** 毎週月曜朝に前週分を集計、ダッシュボード更新＋要約メール配信。

**データソース：** ⑥注文・行動履歴の蓄積 ＋ ⑦KPI算出前処理

---

### ビュー C：月次（中期の関係性を見る）

**先方第2層：関係KPI（月次）を中心に、施策効果が会員の関係性に定着しているか見る層。**

| 項目 | 出典 | 表示形式 |
|---|---|---|
| **月1回以上閲覧会員比率** | 先方KPI（関係KPI） | KPIカード（現値＋前月比＋12ヶ月スパークライン） |
| **未注文週→翌週復帰率** | 先方KPI（関係KPI） | KPIカード（同上） |
| **関係KPI 2系列 12ヶ月チャート** | 先方KPI（関係KPI） | 2軸折れ線（閲覧比率×復帰率） |
| コース別月次分布（上位5） | 業界標準補完 | テーブル（構成比＋前月比＋12ヶ月スパークライン） |
| 分析所見（1段落） | 改善推進担当 | 次月の打ち手仮説を短く |

**出し方：** 毎月第1営業日に前月分を集計、ダッシュボード更新＋レポート配信。

**データソース：** ⑥注文・行動履歴の蓄積 ＋ ⑦KPI算出前処理

---

### ビュー D：四半期（長期トレンドの確認）

**先方第3層：結果KPI（四半期〜）を中心に、長期的な改善の着地を確認。**

| 項目 | 出典 | 表示形式 |
|---|---|---|
| **平均注文頻度 / 四半期** | 先方KPI（結果KPI） | KPIカード（現値＋前Q比＋過去Qスパークライン） |
| **実質離脱数（8週非アクティブ）** | 先方KPI（結果KPI） | KPIカード（同上） |
| 長期トレンドチャート | 先方KPI（結果KPI） | 棒グラフ（当四半期）＋前年同Q点線オーバーレイ |
| リテンション・コホート | 業界標準補完 | 加入月別×経過月数のヒートマップ（濃緑=高継続／赤橙=低継続） |

**出し方：** 四半期末に前3ヶ月分を集計、詳細レポートとして配信。

**データソース：** ⑥注文・行動履歴の蓄積 ＋ ⑦KPI算出前処理

## 重要度ランクの適用（刈り込み前後）

初版では各ビューに多数の補完項目を並べていたが、S/A判定に絞って B判定項目は削除。

| ビュー | 削除した補完項目 | 残した補完項目 |
|---|---|---|
| 日次 | 運用品質モニタ（正規化エラー・ログイン異常） | FC別アラート、締切コース別 |
| 週次 | コース別内訳（月次へ移動）、リテンションコホート（四半期へ一本化） | ファネル、FC別上位5＋アラート |
| 月次 | 平均注文額／月次アクティブ会員数／新規会員／復帰会員（B判定で削除） | コース別月次分布（週次から移動） |
| 四半期 | 顧客生涯価値推計（B判定で削除）、継続率別カット | リテンションコホート |

## バックエンドPDF 8カテゴリとの関係整理

| カテゴリ | ダッシュボードでの役割 |
|---|---|
| ①注文データ受信・正規化 | 参照範囲外（運用品質監視を本ダッシュボードでは省略） |
| ②注文状態・締切管理 | 日次：締切コース別／受付状況の元 |
| ③注文集計ロジック | 日次：前日注文件数／FC別集計の元 |
| ④需要確定・調整補助 | 参照範囲外（運営向けではない） |
| ⑤出荷・仕分け向けデータ生成 | 参照範囲外（現場オペ向け） |
| ⑥注文・行動履歴の蓄積 | **全ビューのデータソース主** |
| ⑦KPI算出前処理 | **ダッシュボード直前の計算層** |
| ⑧管理・運用向け参照処理 | **ダッシュボードのUI・参照層そのもの** |

既存ベンダー取りやめ方針のため、⑥⑦⑧領域はリバプレ側で設計・実装する前提（要確認：既存データ構造の引き継ぎ範囲）。

## 提供物

- **ダッシュボード本体**（静的Webアプリ／軽量SVGチャート、外部BIツール不要）。ブラウザで開く／Notion埋込／スクリーンショットをpptx添付など、先方の運用に合わせ展開可
- 日次／週次／月次／四半期のメール・Slack通知サマリ（閾値超過時を含む）
- **指標定義書**（先方既定義の6KPI＋補完項目の計算定義・データソース対応表）
- 運用引き継ぎ資料（運営担当者が自走できるレベル）

### 段階的リリース

- **第1段階（1〜2ヶ月）：** 先方既定義6KPIを週次／月次／四半期ビューに配置。基本のチャート・カード・テーブル
- **第2段階（3〜4ヶ月）：** 日次ビュー＋施策注釈機能＋FC別アラート＋ファネル。運営監視と改善推進の両用途に対応
- **第3段階（5〜6ヶ月）：** リテンションコホート＋長期トレンド比較＋分析所見の定着。運営の自走化を補助

## 対内補足（対外には書かない）

- **川平さんから なむ への「テキスト媒体でいい」の指示は、提案書の書き方（テキストで記述してよい）についての指示であって、ダッシュボード自体の表示媒体への指示ではない。** ダッシュボードの媒体は提案側の設計判断。モックアップでは分析用Webとした。
- **実装軽量化の意図：** 静的HTML + 軽量JS（外部ライブラリ不使用でSVG直描画）で成立する設計にしてある。モックアップ参照。運用コストは低く、先方の既存インフラ側に負荷をかけない。
- **データソース側の前提：** ⑥⑦領域が既存ベンダー取りやめで引き継げる前提だが、データ層の引き継ぎが成立しない場合は工数が大きく変わる。**事前調査段階の必須確認項目**。
- **先方既定義6KPI は絶対に外さない：** xlsx で明示的に定義済みなので、勝手に差し替えず必ず含める。業界標準項目は "補完" の位置付け。
- **S/A で絞った結果、実装スコープが当初想定より小さい：** その分、金額提示の初期構築費用は抑制可能。運用段階で項目追加要望が出たら月額保守内で吸収する立て付けが自然。
- **ヒアリング精緻化ポイント：** 運営が実際に毎朝／毎週どのビューを開くか、アラートの閾値、配信先の顔ぶれ（運営企画部・各FC統括・etc）は事前調査で確定。
