## 13枚目（改善速度を高める運用サイクル設計）

**スライド見出し**：改善速度を高める運用サイクル設計 〜なぜPDCAが止まるのか〜

```
【現状の課題】
・改修依頼が規模にかかわらず同じ手順に流れ、軽微な改善も都度見積・長期化。
・結果、「仮説→小改修→数値確認→次」のサイクルが回らず、学びが溜まらない。
・障害対応と改善の発案が同じ列に並び、仮説検証の速度が出せない。

【本提案での運用サイクル設計】

① 軽微改修サイクル（本提案の中心）
→画面の調整・文言修正・計測用タグの設置・案分け検証・小さな導線改善
　※週次〜月次で回す短サイクル。保守運用と改善推進を一体で引き受ける。

② 計画的改修サイクル
→中規模の機能追加・改善。要件整備と優先順位付けを改善推進側で主導。
　※従来型の要件→実装→リリース。短サイクルと併走。

③ 基盤改修・全面入れ替え（対象外）
→大規模な基盤構成変更・ゼロからの新規開発
　※基本方針（全面入れ替えは行わない）と整合。

【軽微改修サイクルで何が変わるか】
・判断の主導権：ヨシケイ側の仮説起点で着手できる。
・速度：都度見積ではなく、月額枠内で優先度を入れ替えながら回す。
・学習の蓄積：数値に基づく「やる／やらない」判断が毎月積み上がる。
```

**[図解案]**：3つの同心円／中央「軽微改修サイクル」を最大・強調表示／外側「計画的改修」／外縁破線で「基盤改修＝別議題」

### 補足（資料作成担当向け）
- このスライドが提案全体の肝。「改善推進と保守運用を一体で担い、PDCAを回せる状態を作る」という立ち位置が一目で伝わるよう、中央の「軽微改修サイクル」を色・大きさで強調。
- 既存開発委託先（スーパーネットデジタル）は名指ししない。役割分担の具体は 15枚目／18枚目で扱う構成とし、本スライドでは"規模×速度差"の設計を主役にする。
- 「都度見積・長期化でPDCAが回らない」という既存の課題認識と、本提案が直接接続することを矢印等で示せると望ましい。

### 第3版変更補足（対内）
- 旧版は「定常保守＝既存開発委託先／改善保守＝本提案／基盤改修＝対象外」の3区分だった。冒頭から"定常保守は引き受けない宣言"に読める構造を解消するため、**改修規模に基づく切り分け**に組み替え。
- 「保守は引き受けられる前提で書く」（川平さん指示）と「既存開発委託先の置き換えではない」（`about`）を両立させるため、保守の担い手を名指しで分けるのではなく、サイクル設計として語る。役割分担は 15枚目・18枚目に委譲。

## 18枚目（既存開発委託先との役割分担接点）

**スライド見出し**：バックエンド処理カテゴリ別の関与方針 〜既存開発委託先との共存設計〜

> **設計方針補足（対内向け）：** 本スライドは、先方提供の『楽楽web_バックエンド処理ロジック一覧.pdf』に記載の8区分に対して「どのような考え方で関与可能か」への**正面回答**として設計。既存開発委託先との共存構図（`about` の置き換え禁止方針）を視覚化するのが主目的。
> 第3版で「改善推進の実装責任は表層UIのみ（4・8）」の絞り込み表現を緩め、軽微改修は広く拾える書きぶりに変更。中規模以上の実装と基盤領域が既存開発委託先の主担当であることは維持。

```
【本提案における関与方針】
・基盤〜バックエンド領域は既存開発委託先が主担当として継続。
・改善推進担当は、仮説起点の要件整備／計測設計／軽微改修／画面改善／
　管理画面の実装までを担い、既存開発委託先と併走する。

【区分別の関与方針】

│ # │ 処理区分               │ 主担当         │ 改善推進担当の関与               │
│───│────────────────────────│───────────────│──────────────────────────────────│
│ 1 │ 注文データ受信・正規化 │ 既存開発委託先 │ 計測要件の接続点連携             │
│ 2 │ 注文状態・締切管理     │ 既存開発委託先 │ 計測要件の接続点連携             │
│ 3 │ 注文集計の処理         │ 既存開発委託先 │ 効果指標の算出単位の定義を主導   │
│ 4 │ 需要確定・調整補助     │ 既存開発委託先 │ 判断支援画面の改善を協働         │
│ 5 │ 出荷・仕分けデータ生成 │ 既存開発委託先 │ 関与は必要時のみ                 │
│ 6 │ 注文・行動履歴の蓄積   │ 既存開発委託先 │ 蓄積項目の棚卸し・要件定義       │
│ 7 │ 効果指標の算出前処理   │ 既存開発委託先 │ 算出定義の合意・要件の明文化     │
│ 8 │ 管理・運用向け参照処理 │ 協働           │ 管理画面の改善・軽微改修を実行   │

※表層UI（フロントエンドの画面要素・計測用タグ・案分け検証等）の軽微改修は
　改善推進担当が直接実行。中規模以上のフロントエンド実装は既存開発委託先と協議。

【この分担で得られるもの】
・既存開発委託先は基盤実装の品質保証という本来の強みに集中できる
・改善推進担当は"仮説→要件→軽微改修→計測"の短サイクルを回せる
・責任の重心が明確で、切り分け不明な事案での押し付け合いが起きにくい

【切り分け不明な事案の扱い】
改善推進側の仮説検証（案分け検証・計測用タグの設置等）が
バックエンドの負荷や仕様への影響に波及した場合：
  ① 一次受け：改善推進担当が事象を整理
  ② 要件への落とし込み：既存開発委託先への実装要件に変換
  ③ 実装・反映：影響範囲に応じて既存開発委託先／改善推進側で分担
  ④ 事後レビュー：影響範囲・再発防止を双方で共有
```

**[図解案]**：上段に8区分の表（主担当・関与の位置を色分け）／下段に切り分け不明な事案の4段階の手順図／右端に「主担当=既存開発委託先」「協働/軽微改修=改善推進」の凡例

### 補足（資料作成担当向け）
- 表のセル色：既存開発委託先=青系、改善推進=橙系、協働=中間色。視覚的に**基盤領域は青（既存開発委託先が主担当）**と分かるよう設計。これで「改善推進が全部取りに来る提案ではない」が一目で伝わる。
- 区分8（管理画面）は協働＝中間色。改善推進が実装まで担う領域として視覚化。
- 切り分け不明な事案の手順は矢印図で。「変換・整理工程」を改善推進の付加価値として見せる。
- スマイルスタッフ連携手順（月次の問い合わせ分類データ／四半期のヒアリング）とリリース調整時間枠は**別スライド化候補**。本スライドに詰め込むと情報過多になる。

### 第3版変更補足（対内）
- 第2版の対内判断補足（「8区分中6つは実装責任外／実装責任は表層UI＝4・8のみ」）は対外資料では表現を緩める。軽微改修を広く拾える書きぶりに統一。
- なむ個人の稼働枠（月20時間）との整合は、実務で「軽微」と「中規模以上」の線引きを握って担保する。線引きの具体は月次レビューで都度調整（対内運用）。
- ASP（レシピ）領域は 18枚目の表上では区分に含まれないが、暗黙に「既存開発委託先の主担当（既存ASP保守の実績）」として扱う。対外で明示しない。

## 19枚目（集中するための関与範囲設計）

**スライド見出し**：集中するための関与範囲設計 〜"速く回す"ために絞る〜

```
【考え方】
・改善速度＝仮説を試せる回数。大規模改修に工数を割くほど1サイクルが遅くなる。
・現行基盤を活かし、軽微改修と計画的改修の両輪で成果を積み上げる。

【本提案の対象外】

① ゼロからの新規開発／全面入れ替え
→基本方針と整合。既存資産を活かした改善で成果を出す。

② 基盤構成の全面刷新・データ分析基盤の新規構築
→既存の計測手段（GA等）を最大限活用し、分析基盤の構築自体には予算を割かない。

【前提条件】
・第0段階（事前調査）を経て関与範囲を最終確定。
・既存開発委託先との協力関係：リリース調整・情報共有が可能な状態。
・計測基盤の最低限の整備：タグ設置や行動計測が可能な状態。

【この関与範囲設計で得られるもの】
・短サイクルで仮説を試せる改善運用の定着。
・小さな成功を積み上げる文化の定着。
・既存開発委託先との役割分担による、組織全体としての推進力向上。
```

**[図解案]**：左＝「やること（軽微改修＋計画的改修）」大きめの円／右＝「やらないこと（全面入れ替え／分析基盤新設）」小さめ／下部矢印「絞るから、速い」

### 補足（資料作成担当向け）
- 否定的に見せないために冒頭で「集中するための関与範囲設計」と位置づける。見出しに「対象外」を入れず、前向きな表現で統一。
- 「やらない＝できない」ではなく「やらない＝集中のための選択」という調子で。
- 最後の項目（得られるもの）で着地を前向きにすることで、防衛のためのスライドではなく戦略のためのスライドに見せる。
- 前提条件のうち「既存開発委託先との協力関係」は特に重要。協力関係が前提であることを明示し、対立構図を作らない。

### 第3版変更補足（対内）
- 第2版にあった「24時間365日の緊急対応待機は対象外」「バックエンド基盤・インフラ・DB運用は対象外」「既存ASP保守は対象外」「フロントエンド改善実装の手配自体はリバプレ／既存開発委託先の責務」「月末請求期のリリース作業は対象外」は**対外資料から削除**。
  - 理由：川平さんすり合わせ結果（リバプレ側に別の技術者が在籍・重たい保守は不要）を踏まえ、「保守は引き受けられる前提で書く」指示と衝突するため。
- 残すのは `about` の NG 方針（ゼロからの新規開発／全面入れ替え）と、PDCA が回るための前提条件のみ。
- なむ個人の稼働上限（月20時間）や月末のリリース凍結などは**対内での運用設計**で押さえる。対外資料には書かない。
- 「事前調査を経たうえでの関与範囲確定」は前提条件に維持（第0段階の位置付けと整合）。
