## 13枚目（改善速度を高める運用サイクル設計）

**スライド見出し**：改善速度を高める運用サイクル設計 〜なぜPDCAが止まるのか〜

```
【現状の課題】
・改修依頼が規模にかかわらず同じ手順に流れ、軽微な改善も都度見積・長期化。
・結果、「仮説→小改修→数値確認→次」のサイクルが回らず、学びが溜まらない。
・障害対応と改善の発案が同じ列に並び、仮説検証の速度が出せない。

【本提案での運用サイクル設計】

① 軽微改修サイクル（本提案の中心）
→画面の調整・文言修正・計測用タグの設置・案分け検証・小さな導線改善
　※週次〜月次で回す短サイクル。保守運用と改善推進を一体で引き受ける。

② 計画的改修サイクル
→中規模の機能追加・改善。要件整備と優先順位付けを改善推進側で主導。
　※従来型の要件→実装→リリース。短サイクルと併走。

③ 基盤改修・全面入れ替え（対象外）
→大規模な基盤構成変更・ゼロからの新規開発
　※基本方針（全面入れ替えは行わない）と整合。

【軽微改修サイクルで何が変わるか】
・判断の主導権：ヨシケイ側の仮説起点で着手できる。
・速度：都度見積ではなく、月額枠内で優先度を入れ替えながら回す。
・学習の蓄積：数値に基づく「やる／やらない」判断が毎月積み上がる。
```

**[図解案]**：3つの同心円／中央「軽微改修サイクル」を最大・強調表示／外側「計画的改修」／外縁破線で「基盤改修＝別議題」

### 補足（資料作成担当向け）
- このスライドが提案全体の肝。「改善推進と保守運用を一体で担い、PDCAを回せる状態を作る」という立ち位置が一目で伝わるよう、中央の「軽微改修サイクル」を色・大きさで強調。
- 役割分担の具体は 15枚目／18枚目で扱う構成とし、本スライドでは"規模×速度差"の設計を主役にする。
- 「都度見積・長期化でPDCAが回らない」という既存の課題認識と、本提案が直接接続することを矢印等で示せると望ましい。

### 第4版変更補足（対内）
- 13枚目の文面そのものは第3版から変更なし（運用サイクル設計は方針変更の影響を受けない）。
- 第4版では既存ベンダー取りやめ前提だが、本スライドは規模×速度差の設計の話なので影響軽微。
- 旧版は「定常保守＝既存開発委託先／改善保守＝本提案／基盤改修＝対象外」の3区分だった。冒頭から"定常保守は引き受けない宣言"に読める構造を解消するため、**改修規模に基づく切り分け**に組み替え（第3版時点）。

## 18枚目（対応範囲と業務処理カテゴリ別の関与）

**スライド見出し**：バックエンド処理カテゴリ別の対応範囲 〜本提案でカバーする領域〜

> **設計方針補足（対内向け）：** 本スライドは、先方提供の『楽楽web_バックエンド処理ロジック一覧.pdf』に記載の8区分に対して「どのカテゴリをこちらで対応するか」への**正面回答**として設計。
> 第4版（既存ベンダー取りやめ方針）では、リバプレが単独対応する前提に組み替え。⑥⑦⑧の3カテゴリは **運営向けダッシュボードの主データソース** として明示的に対応範囲に含む。

```
【本提案における対応範囲】
・運営向けダッシュボードの主データソースとなる ⑥⑦⑧ をリバプレ側で巻き取り。
・①②③ の業務処理は、既存業務ロジックを尊重しつつ、ダッシュボード連携・
　運用品質維持の観点で対応。
・④⑤ は業務固有・現場オペ向けで、本提案の対象外。

【カテゴリ別の対応範囲】

│ # │ 処理カテゴリ           │ 対応範囲                                      │
│───│────────────────────────│────────────────────────────────────────────│
│ 1 │ 注文データ受信・正規化 │ ダッシュボード連携／保守運用範囲               │
│ 2 │ 注文状態・締切管理     │ 同上（締切管理は日次ビューに直結）             │
│ 3 │ 注文集計ロジック       │ 同上（前日件数・FC別集計の元）                 │
│ 4 │ 需要確定・調整補助     │ 対象外（業務固有の判断領域）                   │
│ 5 │ 出荷・仕分けデータ生成 │ 対象外（現場オペ向け）                         │
│ 6 │ 注文・行動履歴の蓄積   │ ★ ダッシュボードの主データソース               │
│ 7 │ 効果指標の算出前処理   │ ★ ダッシュボード計算層                         │
│ 8 │ 管理・運用向け参照処理 │ ★ ダッシュボード参照・UI層                     │

【この対応範囲で得られるもの】
・運営向けダッシュボードに必要なデータ層を一気通貫で設計できる
・改善推進サイクル（13枚目）と同じ責任主体で回せるため、
　仮説〜実装〜計測〜判断のループに切れ目が出ない
・現場オペや業務固有の判断領域（④⑤）には踏み込まないことで、
　責任の重心を絞ってリスクを限定する

【データソース引き継ぎについて】
既存システムから ⑥⑦ のデータ層を引き継ぐ範囲は、
事前調査段階で精緻化のうえ確定する。
```

**[図解案]**：8区分の表（対応＝青、対象外＝グレー、★＝橙でハイライト）／右側にダッシュボード章への矢印で「⑥⑦⑧→ダッシュボード」を視覚化／下部に "事前調査で精緻化" の留保バー

### 補足（資料作成担当向け）
- ⑥⑦⑧をハイライトして、**運営向けダッシュボード章（20-22枚目）への接続を明示**。表だけ見せるのではなく、ストーリーとして繋がる構成に。
- 対象外（④⑤）は責任を絞ることのメリットとして肯定的に書く。「踏み込まない＝品質を確保」のトーン。
- データ層引き継ぎは事前調査段階で確定する旨、本スライド下部に留保として明記。

### 第4版変更補足（対内）
- 第3版にあった「主担当＝既存開発委託先／改善推進担当との共存」の表は廃止。
- リバプレ単独対応を前提に「対応範囲」を明示する表に書き直し。
- ⑥⑦⑧をダッシュボードの主データソースとして明示することで、ダッシュボード章（dashboard_proposal.md）と18枚目の連携が成立する。
- なむ個人の稼働枠（月20時間）との整合は、実務で軽微改修と中規模実装の線引きを握って担保（対内運用）。
- 既存ASP（レシピ等）領域は 8区分に含まれないため、本スライドの対象外。事前調査段階で必要に応じて別途扱う。

## 19枚目（集中するための関与範囲設計）

**スライド見出し**：集中するための関与範囲設計 〜"速く回す"ために絞る〜

```
【考え方】
・改善速度＝仮説を試せる回数。大規模改修に工数を割くほど1サイクルが遅くなる。
・現行基盤を活かし、軽微改修と計画的改修の両輪で成果を積み上げる。

【本提案の対象外】

① ゼロからの新規開発／全面入れ替え
→基本方針と整合。既存資産を活かした改善で成果を出す。

② 基盤構成の全面刷新・データ分析基盤の新規構築
→既存の計測手段（GA等）を最大限活用し、分析基盤の構築自体には予算を割かない。

【前提条件】
・第0段階（事前調査）を経て関与範囲を最終確定。
・既存システムからのデータソース引き継ぎが可能な状態（⑥⑦領域）。
・計測基盤の最低限の整備：タグ設置や行動計測が可能な状態。

【この関与範囲設計で得られるもの】
・短サイクルで仮説を試せる改善運用の定着。
・小さな成功を積み上げる文化の定着。
・対応範囲（18枚目）を明示することで、組織全体としての推進力向上。
```

**[図解案]**：左＝「やること（軽微改修＋計画的改修）」大きめの円／右＝「やらないこと（全面入れ替え／分析基盤新設）」小さめ／下部矢印「絞るから、速い」

### 補足（資料作成担当向け）
- 否定的に見せないために冒頭で「集中するための関与範囲設計」と位置づける。見出しに「対象外」を入れず、前向きな表現で統一。
- 「やらない＝できない」ではなく「やらない＝集中のための選択」という調子で。
- 最後の項目（得られるもの）で着地を前向きにすることで、防衛のためのスライドではなく戦略のためのスライドに見せる。
- 前提条件のうち「既存システムからのデータソース引き継ぎ」は事前調査段階で必須確認項目（第4版で重要度上昇）。

### 第4版変更補足（対内）
- 第3版にあった前提条件「既存開発委託先との協力関係」は、既存ベンダー取りやめ方針により撤回。代わりに「データソース引き継ぎが可能な状態」に書き直し（事前調査段階の確認項目）。
- 第2版にあった「24時間365日の緊急対応待機は対象外」「バックエンド基盤・インフラ・DB運用は対象外」「既存ASP保守は対象外」「フロントエンド改善実装の手配自体はリバプレ／既存開発委託先の責務」「月末請求期のリリース作業は対象外」は**対外資料から削除**（第3版で削除済み）。
- 残すのは `about` の NG 方針（ゼロからの新規開発／全面入れ替え）と、PDCA が回るための前提条件のみ。
- なむ個人の稼働上限（月20時間）や月末のリリース凍結などは**対内での運用設計**で押さえる。対外資料には書かない。
