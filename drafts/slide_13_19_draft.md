## Slide 13（本件における"保守"の再定義）

**スライドタイトル**：本件における"保守"の再定義 〜なぜPDCAが止まるのか〜

```
【現状の課題】
・「保守」が一つの塊として扱われ、改修依頼が全て同じフローに流れている。
・障害対応と改善アイデアが同じ列に並び、仮説検証の軽微な改修まで都度見積・長期化。
・結果、「仮説→小改修→数値確認→次」のサイクルが回らず、学びが溜まらない。

【本提案での"保守"の切り分け】

① 定常保守（現ベンダー継続）
→障害対応・予防保守・インフラ運用・セキュリティ対応
　※安定稼働の土台。これまで通り安心して任せる領域。

② 改善保守（本提案の対象）
→仮説検証に伴う軽微改修・UIコピーや導線の調整・計測タグ設置・A/Bテスト分岐
　※「小さく試して数値で判断」を回す領域。

③ 基盤改修（対象外）
→大規模リプレイス・アーキテクチャ変更・フルスクラッチ
　※基本方針（フルリプレイスNG）と整合。

【"改善保守"が他と何が違うか】
・判断の主導権：ヨシケイ側の仮説起点で着手できる。
・速度：都度見積ではなく、週次で優先度を入れ替えながら回す。
・対象範囲：数値で効果が測れる小さな改修に絞る。
```

**[図解]**：3つの円を横並び／中央の「改善保守」をハイライト／下部に「定常保守＝守り」「改善保守＝攻め」「基盤改修＝別議題」の帯

### 補足メモ（デザイナー向け）
- このスライドが提案全体の肝。「受託ベンダーの置き換えではなく、改善推進パートナー」という立ち位置が一目で伝わるよう、中央の「改善保守」を色・サイズで強調。
- 現ベンダー（スーパーネットデジタル）は名指ししないが、「定常保守＝現ベンダー継続」を明示することで、置き換え提案ではないことを担保。
- 「都度見積・長期化でPDCAが回らない」という既存の課題認識と、本提案が直接接続することを矢印等で示せるとベター。

## Slide 18（既存ベンダーとの役割分担インターフェース）

**スライドタイトル**：バックエンド処理カテゴリ別の関与方針 〜現ベンダーとの共存設計〜

> **設計方針メモ（対内向け）：** 本スライドは、先方提供の『楽楽web_バックエンド処理ロジック一覧.pdf』に記載の8カテゴリに対して「どのような考え方で関与可能か」への**正面回答**として設計。同時に、改善保守パートナー（＝片岡さん個人）が**実装責任を持つ範囲を表層UIに限定**することを、表形式で可視化する装置でもある。proposal_critique.md の致命指摘1（オーダーへの正面回答欠落）と指摘3（現ベンダー排除感）への同時対応。

```
【本提案における関与方針】
・各カテゴリの主担当は現ベンダーが継続。改善保守パートナーは"要件側"と
　"表層UI側"で補完的に関与する。
・実装責任の重心は現ベンダーに残し、改善保守は仮説・計測要件・UI改善に集中。

【カテゴリ別関与方針】

│ # │ 処理カテゴリ            │ 主担当   │ 改善保守の関与              │
│───│────────────────────────│─────────│────────────────────────────│
│ 1 │ 注文データ受信・正規化 │ 現ベンダー │ 計測要件接続点の連携のみ    │
│ 2 │ 注文状態・締切管理     │ 現ベンダー │ 計測要件接続点の連携のみ    │
│ 3 │ 注文集計ロジック       │ 現ベンダー │ KPI算出単位の定義をリード   │
│ 4 │ 需要確定・調整補助     │ 現ベンダー │ 人判断UIの表示改善のみ      │
│ 5 │ 出荷・仕分けデータ生成 │ 現ベンダー │ 関与しない                  │
│ 6 │ 注文・行動履歴の蓄積   │ 現ベンダー │ 蓄積項目の棚卸し・要件定義  │
│ 7 │ KPI算出前処理          │ 現ベンダー │ 算出定義の合意・要件の明文化 │
│ 8 │ 管理・運用向け参照処理 │ 改善保守 │ 管理画面UI改善を実装含め担当 │

【この分担で得られるもの】
・現ベンダーは基盤実装の品質保証という本来の強みに集中できる
・改善保守は"仮説→要件→UI改修→計測"の短サイクルを表層で回せる
・責任の重心が明確で、グレー事象での押し付け合いが起きにくい

【グレー事象の扱い】
改善保守側の仮説検証（A/B分岐・計測タグ設置等）がバックエンド負荷
や仕様影響に波及した場合：
  ① 一次受け：リバプレ（改善保守パートナー）が事象を整理
  ② 要件翻訳：現ベンダーへの実装要件に落とす
  ③ 実装・反映：現ベンダーが基盤側で対応
  ④ 事後レビュー：影響範囲・再発防止を双方で共有

→ グレー事象の翻訳・整理工程を改善保守枠に含めることで、
　現ベンダーの負担を増やさず、ヨシケイ側の窓口を一本化する。
```

**[図解]**：上段に8カテゴリの表（主担当・関与ポジションを色分け）／下段にグレー事象の4ステップフロー図／右端に「主担当=現ベンダー」「補完=改善保守」の凡例

## Slide 19（スコープ外・前提条件）

**スライドタイトル**：集中するためのスコープ設計 〜"速く回す"ために絞る〜

```
【考え方】
・改善速度＝仮説を試せる回数。間口を広げすぎると1本あたりが遅くなる。
・「やらないこと」を先に決めることで、「やること」に全リソースを注ぐ。

【本提案のスコープ外】

① 24時間365日のオンコール・緊急時対応
→安定運用は現ベンダーの定常保守に委ねる。必要時は別途協議。

② バックエンド基盤・インフラ・DB運用
→土台領域は触らない。改善保守はフロント〜計測層に集中。

③ フルスクラッチ／全面リプレイス
→基本方針と整合。既存資産を活かした改善で成果を出す。

④ データ分析基盤の新規構築
→既存の計測ツール（GA等）を最大限活用し、分析に予算を割かない。

【前提条件】
・Phase 0（ディスカバリ）を経てスコープを最終確定。
・現ベンダーとの協力関係：リリース調整・情報共有が可能な状態。
・計測基盤の最低限の整備：タグ設置やイベント計測が可能な状態。

【このスコープ設計で得られるもの】
・週次で仮説を試せる改善サイクル。
・小さな成功を積み上げる文化の定着。
・現ベンダーとの役割分担による、組織全体としての推進力向上。
```

**[図解]**：左＝「やること（改善保守）」大きめの円／右＝「やらないこと」小さめの4項目／下部矢印「絞るから、速い」

### 補足メモ（デザイナー向け）
- ネガティブに見せないために冒頭で「集中するためのスコープ設計」と位置づける。タイトルに「スコープ外」を入れず、前向きな表現で統一。
- 「やらない＝できない」ではなく「やらない＝集中のための選択」というトーンで。
- 最後のブロック（得られるもの）で着地を前向きにすることで、防衛スライドではなく戦略スライドに見せる。
- 前提条件のうち「現ベンダーとの協力関係」は特に重要。パートナーシップ前提であることを明示し、対立構図を作らない。
