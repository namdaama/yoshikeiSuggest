## 2. 楽楽web 技術スタック実測（公開画面分析ベース）

### 確認済みスタック
- **フロントエンド**：jQuery + Swiper.js（旧版、v5系） + MicroModal + jquery.calendario.js + object-fit-images polyfill
- **IE互換記述**：`X-UA-Compatible chrome=1`、`ofi.min.js`（古いブラウザ対応を引きずっている痕跡）
- **CSS**：個別ファイル7本を同期読込、バージョン管理は**手動タイムスタンプ**（`?2025102000`）
- **アセット配信**：AWS S3（`yoshikei-it-smile-order-upload` バケット、東京リージョン）
- **サーバーサイド**：直接確認できず（おそらくPHP/Javaベース、SPA化されていない）

### レシピ連携（重要な発見）
- **Classic ASP（Microsoft）でレシピサイトが動作**：`https://www2.yoshikei-dvlp.co.jp/webodr/apl/10/101508_D.asp`
- ASPは1996〜2000年前後の技術。2026年現在、保守可能な技術者は極めて限定的
- 楽楽web（現行）⇄ レシピ（ASP）は URL パラメータ経由で連携

### FC別パラメトリゼーション
- URL パラメータに `FC=TOY` 等（フランチャイズコード）
- 価格・取扱商品・締切が**64FC単位で異なる可能性**
- 改修時のテスト面が大きい原因

### デプロイ・運用
- 2025年10月にデプロイ形跡（CSSバージョンから）
- **運用停止中の機能がHTMLに残存**：検索フォーム、景品プレゼント・特別販売バナー、今週の人気ランキング
  - → 改善機会 or 技術負債（両面あり）

## 3. 商品カタログ規模と構造

### 1週分メニューページの構成
- 夕食献立 **19コース**（スマイリー／スタンダード／バランス／カット／ファン／タイムリー／プレミアム／カジュアル／バリエーション／クイック／シンプル／ベジ／フリー／人気／コレdo!-2/3/4／セレクト／特集）
- 各コース：月〜土（6日）× 1〜4人サイズ × 楽々/通常モード
- まとめセット（5日間／6日間）
- 特別販売（食彩健美など、30+アイテム）
- アラカルト（朝食／レトルトごはん）
- **推定総SKU数：500〜1000／週**
- 画面上の **Swiperインスタンス数：26個**

### 反復構造の重要な性質
- レンダリングHTMLは巨大だが、**テンプレート本体は小さい**（50〜100行程度と想定）
- **1箇所の変更がカタログ全体に波及する構造**
- これは改善保守パートナー提案にとって有利：
  - 軽微改修の費用対効果が極大
  - A/B分岐をテンプレレベルで仕込める
  - 計測タグ1箇所で全カード計測可能
  - PDCA「小さく試して数値で判断」と相性が良い

## 4. 改修ボリューム感（反復構造を踏まえた見積）

| 改修種類 | 実質所要 | 備考 |
|---|---|---|
| 文言・コピー変更（テンプレ起因） | 30分〜1h | テンプレ1箇所でカタログ全体に波及 |
| ボタンコピー統一変更 | 30分〜2h | 同上 |
| 計測タグ追加（テンプレ埋込） | 1〜2h | 1箇所で全カード発火 |
| UIポップアップ追加 | 3〜8h | モーダル4種と競合確認必要 |
| A/B分岐実装 | 5〜10h | テンプレレベルで分岐可能 |
| 商品カード全面改修 | 15〜30h | コード変更は軽いがQA面が広い |
| ランキング機能復活 | 30〜60h | データ生成側の実装が必要 |
| 新機能追加（中規模） | 40〜80h | ASP側との連携絡むと+α |

### QA工数の扱い
- コード変更が軽くても、**全コース×全曜日×全モード×全FC**でデグレ確認必要
- jQuery時代のため自動テスト不十分と想定
- 26 Swiper に影響しないか手動確認

## 5. 先方の組織・商流コンテキスト

### ステークホルダーマップ
- **ヨシケイ開発**：発注側本部
- **スーパーネットデジタル**（現ベンダー）：定常保守・改修
- **リバプレ**（川平さん所属想定）：元請候補
- **片岡さん**：改善推進パートナー候補（個人受注・副業）
- **スマイルスタッフ**：現場活動・顧客接点
- **競合選定あり**（複数候補）

### 川平さんの技術リテラシー
- ソフトウェアエンジニアリングの技術リテラシーは**高くない前提**
- 用語の使い方が緩く、「バックエンド」を「管理画面UI」の意味で使う場面あり
- 先方の真意を川平さん経由で聞く際はノイズが乗る前提
- 一次資料（先方提供の文書）を優先する方針

### ヨシケイ側の明示方針（`about` より）
- 現在のWebサイト基盤・業務構造は活用する
- フルリプレイス・フルスクラッチは行わない
- 改善の主導権（企画・判断）のみを外部に補完的に委ねる
- **受託開発ベンダーの置き換えではない**
- NG方針：フルスクラッチ／丸投げ／EC構築実績のみでのベンダー選定

### 先方の水面下シグナル（推定）
- 課題セクションで現ベンダー批判が強めに書かれている
- 「EC構築実績のみでの選定NG」は新規ベンダー評価軸を示唆
- ただし**ASP（レガシー）保守を引き受けられる代替ベンダーが事実上存在しない**ため、現ベンダー置き換えは技術的に困難

### 先方の運用側リスク（`about` の書きぶりから）
- 課題を外部（現ベンダー）に帰している書き方
- 「改善の主導権」を外部に委ねる願望 → 内部プロダクトオーナー不在を示唆
- 発注側の要件粒度・優先順位付け体制が不明
- → 新パートナーに変えても発注側の運用変革がないとPDCAは回らない可能性

## 6. 片岡さん側の制約

### 稼働上限
- **副業前提、月20時間まで**
- 実質生産時間：定例・資料作成・コンテキストスイッチを除いて**月9〜10時間**
- 実装を抱えるのは物理的に不可能
- **アドバイザリー＋要件定義までが限界**

### 役割の現実解
- 企画・判断・KPI監修・月次レビュー
- 仮説起点の優先順位付け
- UI/UX改善の要件定義
- 既存ベンダーへの要件翻訳
- A/Bテスト設計のレビュー
- → **実装は別リソース**（リバプレ or 現ベンダー）必須

### Claude Code 支援の現実的な効果
- ドキュメント作成・議事録・リサーチで ~3〜4h/月 短縮
- **片岡さんは楽楽webコードベースへのアクセスを持たない**ため、コード生成の恩恵は発生しない
- 「実装抱える」を可能にする効果はない
- アドバイザリー役の余裕を作る方向に効く

## 7. 対価レンジの根拠

### 業界相場（2026年4月時点／公開情報）
- 一般ECサイト保守：月5〜80万円
- EC改善パートナー契約：月10〜50万円、6ヶ月合計150〜250万円目安
- 中堅〜独立系エンジニア人月単価：70〜250万円（時間単価4,500〜16,000円）

### ヨシケイ向けの妥当レンジ
- **月額固定：30〜50万円** ＋ スポット別建て
- 個人稼働月20h × 時間単価 15,000〜25,000円 の逆算
- リバプレ→ヨシケイの請求は **1.3〜1.5倍**（40〜75万円）想定
- 金額提示は**レンジのみ／ディスカバリ後に精緻化**を明示

### なぜ相場上限域ではなく中位に設定するか
- 個人受注のため稼働上限が明確（20h）
- 実装を抱えない立て付けのため工数責任が軽い
- テンプレ共通化で改修の費用対効果が高く、単価を無理に引き上げなくても先方価値が出る
- 競合選定で「高すぎる」判定を避ける

## 8. 業界フェーズと提案訴求軸

### 市場データ
- 食品宅配市場 2024年度 **2兆6,380億円**（+1.7%、矢野経済研究所）
- コロナ特需後の **緩成長フェーズ**
- 競争軸は **会員数拡大 → ARPU×継続率** にシフト

### 成功しているWeb施策の共通項（公開事例ベース）
- **離脱直前の行動変容を狙う軽量機能**
  - カスタマイズ忘れアラート（オイシックス）
  - 属性別クーポン表示
  - 購入数連動割引（nosh）
- **単発キャンペーン依存 → リテンション重視**への移行

### 解約要因（MMD／クロス・マーケティング調査）
- 価格／配達時間／商品の繰り返し／鮮度トラブル／カタログ実物ギャップ
- 後者のうち「注文の退屈さ・意外性の欠如」は**Web側UX改善で効く領域**

### 提案書訴求軸（本件の核）
1. 業界フェーズ俯瞰：「会員数拡大 → ARPU×継続率」の文脈
2. 先方既定義の 3層KPI（行動→関係→結果）は業界成功パターンと整合
3. **楽楽webのテンプレ共通化構造が軽微改修×PDCAと相性が良い**（技術的裏付け）
4. 小改修×PDCAが実効解（大規模リプレイスではない）
5. 個人／小チームだからこそ短サイクルが回せる（ベンダー型との対比）
6. 現ベンダーとの共存設計（置き換えではない＝ASPレガシー保守を引き受けない合理的立場）
